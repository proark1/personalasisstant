# Cutover runbook — off Supabase.com → self-hosted on Railway Postgres

Decision (chosen by the project owner): **real cutover** onto Railway Postgres,
keeping `@supabase/supabase-js` working by self-hosting the open-source
Supabase-compatible services on Railway, and **converting the 103 edge
functions to Node**.

This avoids hand-rewriting ~850 client call sites. The frontend changes by a
single env repoint; the work is infrastructure + RLS + the edge-function port.

## Why this shape

`supabase-js` is just a client for three OSS servers. Self-host them on Railway,
in front of your Railway Postgres, and the app keeps working:

| App surface (measured) | Count | Handled by |
|---|---:|---|
| `.from(...)` table calls | 738 | **PostgREST** |
| `.rpc(...)` (is_admin, kg_neighborhood, kg_top_entities) | 5 | **PostgREST** `/rpc` |
| `supabase.auth.*` (login/signup/session) | 22 | **GoTrue** (or Auth.js — see decision) |
| Realtime (`.channel`/`postgres_changes`) | 90 | **Realtime** server |
| `.storage.from()` direct | 0 | n/a (uploads go via edge functions) |
| Edge functions invoked from the app | 57 of 103 | **Node services** (Phase 3) |

## Services to provision on Railway

1. **Postgres** (you have this). Apply the bootstrap in order — now four files:
   ```bash
   export DATABASE_URL="postgres://...railway.app:.../railway"
   ./db/migration/apply-bootstrap.sh   # 00 → 01 → 02 → 03_rls_policies.sql
   ```
   `03_rls_policies.sql` is generated by `db/migration/squash-schema.ts` and
   restores the 756 policies + 223 RLS toggles the squash strips, plus the
   `anon`/`authenticated`/`service_role`/`authenticator` roles and
   `auth.uid()`/`auth.role()`/`auth.jwt()` helpers that read the JWT.

2. **PostgREST** (`postgrest/postgrest` image). Env:
   - `PGRST_DB_URI` = the Postgres connection string
   - `PGRST_DB_SCHEMAS=public`
   - `PGRST_DB_ANON_ROLE=anon`
   - `PGRST_JWT_SECRET` = a long random secret (shared with the auth service)

3. **GoTrue** (`supabase/gotrue` / `supabase/auth` image):
   `GOTRUE_DB_DRIVER=postgres`, `DATABASE_URL`, `GOTRUE_JWT_SECRET` = **the same
   secret as `PGRST_JWT_SECRET`**, `GOTRUE_SITE_URL`, SMTP vars for email.
   After it boots once, run `db/bootstrap/04_gotrue_bridge.sql` (see below).

4. **Realtime** (`supabase/realtime` image) — env per its docs; same JWT secret.
   Reads the WAL, so Postgres needs `wal_level=logical`.

5. **A gateway** so the SPA hits one origin. Either a tiny reverse proxy
   (Caddy/Nginx) routing `/rest/v1/*`→PostgREST, `/auth/v1/*`→GoTrue,
   `/realtime/v1/*`→Realtime (this is exactly what Supabase's Kong does), or
   Railway's networking. Point `VITE_SUPABASE_URL` at this gateway.

### Repoint the app (the only frontend change)

```
VITE_SUPABASE_URL=https://<your-railway-gateway>
VITE_SUPABASE_PUBLISHABLE_KEY=<anon JWT signed with the shared secret, {"role":"anon"}>
```

The "anon key" / "service key" are just JWTs signed with the shared secret
carrying `{"role":"anon"}` / `{"role":"service_role"}`. Mint them once.

## Auth issuer — GoTrue (decided)

GoTrue self-hosted, so all 22 `supabase.auth.*` calls keep working unchanged.
GoTrue owns its own `auth.users`; the app's FKs point at `public.users` (squash
rewrite). `db/bootstrap/04_gotrue_bridge.sql` reconciles the two:

- mirrors `auth.users.id`/email → `public.users` (so app FKs resolve), and
- recreates the "make a `public.profiles` row on signup" trigger the squash
  stripped (the app reads profiles in 42 places).

GoTrue builds the `auth` schema on first boot, so **`04` must run AFTER the
GoTrue service has started once** — it is NOT part of `apply-bootstrap.sh`:

```bash
# after GoTrue is up:
psql "$DATABASE_URL" -v ON_ERROR_STOP=1 -1 -f db/bootstrap/04_gotrue_bridge.sql
```

`auth.uid()` (in `03_rls_policies.sql`) reads the JWT `sub` that GoTrue mints,
so RLS lines up with `public.users.id` = `auth.users.id`.

## Phases & test gates

| Phase | Scope | Gate |
|------:|-------|------|
| 1 (this PR) | Reconstitute RLS (`03`), apply script, runbook | `apply-bootstrap.sh` runs clean on a fresh Railway PG |
| 2 | Stand up PostgREST + auth + gateway; repoint env | login + a `.from('tasks')` read/write work against Railway |
| 3 | Convert 103 Deno edge functions → Node services | each function's callers work; AI/calendar/email green |
| 4 | Stand up Realtime; verify the 90 subscriptions | live task/chat updates work |
| 5 | Storage (edge-function uploads → S3/R2/Railway volume) | document/recording uploads work |
| 6 | Decommission Supabase | nothing points at `*.supabase.co` |

Until Phase 2's repoint, the app still talks to Supabase.com — so each phase is
independently deployable and testable.
