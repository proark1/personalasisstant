# OneBrain Assistant Implementation Plan

Date: 2026-07-08

## Objective

Build a business personal assistant product that works as a module on top of OneBrain. OneBrain is the canonical data layer for durable business data, memory, permissions, retrieval, and audit-of-record. The assistant is the work interface across web, voice, and Telegram: briefings, inbox triage, calendar planning, follow-ups, meeting prep, drafts, and approved actions.

The first deployment target is Railway for fast testing. The architecture must stay portable so the same product can later move to a Yameena server, EU VM, Docker Compose, Kubernetes, or another GDPR-compliant environment without a rewrite.

## Non-Negotiable Decisions

1. OneBrain owns all durable business data, memory, permissions, retrieval, and audit-of-record.
2. The assistant owns only operational state needed to run the product: jobs, leases, locks, outbox rows, idempotency keys, sync cursors, provider subscriptions, encrypted token storage or secret references, caches, and action execution state. This operational state must not become a second source of truth for business memory.
3. Every durable business record has account, space, purpose, provenance, classification, retention, and audit metadata. Operational records must carry enough account, user, action, job, and correlation metadata to be auditable and safely erased or exported where required.
4. Railway is a deployment target, not a product dependency.
5. Services are Dockerized and configured with environment variables.
6. All providers sit behind interfaces: AI, voice, email, calendar, queue, scheduler, storage, notifications, observability.
7. The assistant follows the current OneBrain architecture direction: Python/FastAPI remains the authoritative backend style, Next.js/React/TypeScript is the product UI shell, and FastAPI OpenAPI contracts generate or drive typed TypeScript clients.
8. All external content is untrusted data, never instructions. Email bodies, web content, calendar descriptions, attachments, Telegram messages, and tool output must pass through normalization, extraction, policy checks, and action validation before they can influence a tool call.
9. Prompt-injection defense, egress controls, approval gates, action validation, idempotency, secrets, audit, observability, and scheduler infrastructure are foundation requirements, not late-phase hardening.
10. Human approval is required for irreversible external sends, deletes, forwards, and external calendar writes in V1.
11. Actions are durable typed records with state transitions, idempotency keys, policy decisions, provenance, and cross-channel deduplication.
12. Every external side effect goes through a transactional outbox and idempotent executor. No provider write should be performed directly from an HTTP request, model output, or Telegram callback.
13. OAuth permissions follow least-privilege scope tiers. Users can connect read-only first, then explicitly grant draft/write/send/calendar-write permissions when needed.
14. OAuth refresh tokens, Telegram bot tokens, webhook secrets, and provider API keys are encrypted from day one using a `SecretProvider`. Durable records store secret references, never raw secrets.
15. Email and calendar sync uses push-first change notifications where practical, with renewal jobs, delta/history cursors, polling fallback, and reconciliation after missed events.
16. Briefs and meeting prep are generated by timezone-aware scheduled jobs, not only when the user opens the app.
17. Observability is part of the product: every proposal, approval, model call class, sync step, and external action has logs/traces/audit correlation.
18. Voice is a primary interface from the beginning, but MVP voice is read-only and proposal-only. Voice can ask, summarize, and create action proposals; it cannot execute irreversible external actions.
19. Telegram is a primary communication layer for private assistant chat, push-style updates, briefings, reminders, and approvals.
20. Telegram is implemented as one `NotificationChannel`, so native push, email, Slack, WhatsApp, and other channels can be added later.
21. The first UI is web-based, but it must be mobile-first and ready to port to Android/iOS later.
22. The UI stays simple: Today, Inbox Review, Follow-Ups, Calendar Plan, Assistant, Settings.
23. Autonomy is earned by action type, sender, space, category, policy, and user acceptance history. Narrow autonomy is post-MVP unless the action is reversible/internal and explicitly whitelisted.
24. The user should reach the most important daily work in one click from Today. Two clicks are acceptable only for exceptions, details, editing, settings, sensitive approvals, or account/provider management.
25. The product behaves like a real personal assistant: it structures the day, protects attention, remembers important people and priorities, learns from feedback, and explains recommendations in plain language.
26. Settings must be searchable and grouped by plain user intent, not by technical provider categories. The user should think "what do I want to change?", not "which subsystem owns this?"
27. The product needs a quiet, work-focused design system before UI implementation: dense but calm, easy to scan, strong hierarchy, restrained color, and brand-aligned with assad-dar.de. Main and large surfaces should stay light; dark colors are for text, fine UI details, and small emphasis only.
28. Calendar planning must protect focus time, not only find empty slots. The assistant should detect overloaded days, preserve deep-work blocks, and suggest moving or batching lower-priority work.

## Recommended Initial Stack

Use the stack that matches the current OneBrain direction and keeps portability high:

- Product UI: Next.js/React/TypeScript, matching `onebrain-web`.
- Backend/orchestration: Python/FastAPI, matching the OneBrain core API.
- Worker: Python worker process sharing the same backend package, domain modules, provider interfaces, OneBrain SDK, and Pydantic contracts as the API.
- Type boundary: FastAPI OpenAPI schema drives generated or typed TypeScript clients for the web app.
- Codebase shape: modular monolith, not microservices. One backend codebase/image runs as two entrypoints: `assistant-api` and `assistant-worker`.
- Operational database: Postgres for actions, action transitions, idempotency keys, transactional outbox, sync cursors, provider subscriptions, encrypted token storage or secret references, job bookkeeping, policy rows, and operational audit correlation.
- Queue/scheduler: Redis-backed queue, scheduler, leases, delayed jobs, retries, distributed locks, and dead-letter handling behind `QueueProvider`/`SchedulerProvider`.
- Data of record: OneBrain service APIs and OneBrain database own durable business data, memory, permissions, retrieval, and audit-of-record.
- Packaging: Docker image for backend API/worker plus a Dockerized Next.js web app.
- Local parity: Docker Compose with web, API, worker, Postgres, Redis, and configurable OneBrain endpoint.
- Prototype hosting: Railway web service, API service, worker service, Postgres, and Redis.
- Future hosting: Yameena/self-hosted Docker Compose first, then optional Kubernetes.
- Secrets: `SecretProvider` from day one, with Railway-friendly storage first and a clean path to managed KMS/secret manager later.
- Observability: structured logs, traces, metrics, audit correlation, model/provider cost tracking, job status, and outbox/action execution visibility from day one.
- AI: tiered `LLMProvider`/model router. Cheap classifier or embeddings for triage, stronger model for drafts, planning, and ambiguous cases, with provider swap support for EU-hosted or self-hosted models later.
- Voice: Gemini first for speed, behind provider adapters so STT/TTS can change later for cost, EU processing, quality, or customer-specific compliance.
- Telegram: first concrete `NotificationChannel` using provider-neutral notification/action records.
- Mobile readiness: responsive Next.js/PWA principles now; keep a Capacitor-style native wrapper path open later.

The framework decision for V1 is no longer open: use the OneBrain hybrid pattern unless OneBrain itself changes direction.

## Final Architecture Decision

Build a modular monolith with one backend codebase and two backend entrypoints:

```text
assistant-web       Next.js / React / TypeScript
assistant-api       Python / FastAPI
assistant-worker    Python worker using the same backend package
postgres            operational state
redis               queue, scheduler, locks, retries, dead-letter jobs
onebrain            durable business data, memory, permissions, retrieval, audit-of-record
```

The API stays thin. It authenticates, runs OAuth, receives provider webhooks, serves UI state, records approvals, and enqueues work. The worker is where the assistant actually happens: sync, triage, brief generation, meeting prep, prompt-injection screening, proposal creation, Telegram delivery, subscription renewal, reconciliation, and approved action execution.

No LLM call or external provider write should be required to complete an ordinary HTTP request. Request handlers write operational state and enqueue work. Workers process durable jobs and write results through OneBrain and the operational database.

Use a transactional outbox for external side effects. State changes and outbox rows must be committed together in Postgres; workers relay outbox rows with idempotency keys. This is the mechanism behind "no double-send" and "no lost action after a crash."

OneBrain's current architecture is the model to follow: Python/FastAPI keeps authority over security, policy, data access, retrieval, privacy, and service boundaries; Next.js renders the product shell through typed API clients and must not reimplement authorization or durable business rules.

## System Shape

### Deployable Services

- `assistant-web`: Next.js user-facing app with voice/text assistant, Today, Inbox Review, Follow-Ups, Calendar Plan, Settings, onboarding, and typed API client boundary.
- `assistant-api`: Python/FastAPI orchestration API for auth/session handoff, OAuth, webhooks, UI state, approvals, and enqueueing jobs.
- `assistant-worker`: Python worker process using the same backend package as the API. Runs sync, triage, brief generation, follow-up extraction, prompt-injection screening, Telegram delivery, voice jobs, subscription renewal, reconciliation, outbox relay, and approved action execution.
- `postgres`: operational state store for action state, outbox, jobs, locks/leases, idempotency keys, sync cursors, provider subscriptions, policy rows, encrypted token storage or secret references, and operational audit correlation.
- `redis`: queue, delayed jobs, scheduler wakeups, distributed locks, retries, and dead-letter queue.
- `onebrain`: external or sibling OneBrain API/core that owns durable business data, memory, permissions, retrieval, privacy, and audit-of-record.

### Worker Modules

These are modules inside `assistant-worker`, not separate V1 microservices:

- `telegram_gateway`: private Telegram bot chat, inbound commands, outbound updates, approval callbacks, and optional voice-message intake.
- `connector_sync`: Gmail/Outlook/Calendar sync, labels/categories, drafts, subscription renewal, cursor/delta management, reconciliation, and approved writes.
- `briefing_jobs`: morning brief, day refresh, end-of-day recap, weekly report, and meeting prep.
- `followup_jobs`: commitment extraction, waiting-on detection, overdue checks, and delegated-task refresh.
- `voice_jobs`: transcription, voice turn handling, spoken response generation, and transcript persistence.
- `action_executor`: approved action execution, idempotency enforcement, final policy check, provider response capture, and audit writes.
- `security_pipeline`: untrusted content normalization, hidden text stripping, prompt-injection and egress checks, and quarantined structured extraction records.
- `scheduler_jobs`: timezone-aware recurring and delayed jobs for briefs, meeting prep, sync reconciliation, provider subscription renewal, retry backoff, and quiet-hour delivery.
- `outbox_relay`: reads transactional outbox rows and performs idempotent external side effects.

### Core Interfaces

- `BrainClient`: OneBrain capture, retrieval, policy checks, memory, tasks, action records.
- `LLMProvider`: classify, extract, summarize, draft, plan, with model tiering and cost/rate-limit controls.
- `ModelRouter`: chooses cheap/strong/EU/self-hosted models by task, sensitivity, latency, cost, and deployment policy.
- `SpeechToTextProvider`: speech to transcript.
- `TextToSpeechProvider`: assistant response to audio.
- `NotificationChannel`: send, receive, preference-check, quiet-hours-check, delivery receipt, and action callback support across Telegram first, then web/native push/email/Slack/WhatsApp later.
- `TelegramChannel`: first concrete `NotificationChannel`, including private assistant chat, outbound notifications, inbound commands, approval callbacks, and optional voice-message intake.
- `EmailConnector`: read, label, draft, send-after-approval.
- `CalendarConnector`: availability, event draft, write-after-approval.
- `QueueProvider`: background jobs.
- `SchedulerProvider`: recurring jobs.
- `OutboxProvider`: transactional outbox writes, relay leasing, idempotency, retries, and dead-letter handling.
- `OperationalStore`: Postgres-backed operational state for actions, sync, jobs, policies, and provider bookkeeping.
- `SyncProvider`: provider-specific push subscriptions, webhook verification, cursor/delta storage, polling fallback, reconciliation, and renewal.
- `SecretProvider`: encrypted storage/retrieval for OAuth refresh tokens, Telegram bot tokens, webhook secrets, and provider keys.
- `ContentSanitizer`: HTML normalization, hidden text removal, invisible Unicode handling, attachment metadata extraction, and safe text conversion.
- `InstructionFirewall`: treats external content as data only, detects injection patterns, separates instructions from content, and blocks unsafe egress paths.
- `ActionPolicyEngine`: evaluates autonomy policy, approval requirements, recipient safety, sensitivity, reversibility, channel eligibility, and user preferences.
- `ActionStore`: typed action state machine with idempotency keys and cross-channel deduplication.
- `StorageProvider`: temporary audio/file objects.
- `ObservabilityProvider`: logs, metrics, traces, audit correlation.

- Priority scores and priority explanations.
- Daily briefs.
- Meeting records and action items.
- Contact profiles.
- Relationship intelligence: VIPs, clients, internal teammates, private contacts, preferred tone, last interaction, open loops, and follow-up expectations.
- Preference memories.
- Voice transcripts and voice-derived action proposals.
- Telegram chat bindings and Telegram message provenance.
- Notification events and delivery preferences.
- Settings intent preferences: notification rules, channel defaults, language, voice preferences, provider visibility, privacy choices, and autonomy comfort level.
- Autonomy policies.
- Tone profiles.
- Action proposals, approvals, executions, failures, policy decisions, source references, and audit-of-record.
- Exact approved action snapshots and content hashes for email/calendar writes.
- Assistant feedback.
- Teaching signals such as "not important", "make VIP", "never notify", "shorter tone", "private not work", and "good suggestion".
- Uncertainty events.
- Prompt-injection/security decisions.
- Model usage, model tier, cost class, latency class, and provider error metadata.
- Provider health and graceful-degradation events.
- Append-only action audit entries.

Each API should support purpose-scoped capture and retrieval. The assistant should never fetch broad data and ask the model to obey privacy rules. OneBrain must filter before context reaches AI.

Raw secret values do not belong in OneBrain records. OneBrain stores secret references, scope grants, ownership, purpose, and audit metadata. Secret values live behind `SecretProvider`.

Action execution state is split deliberately: Postgres stores the operational state machine, leases, outbox rows, retries, and idempotency keys needed to execute safely; OneBrain stores action provenance, approval facts, source references, final execution facts, and audit-of-record.

## Security And Agent Safety Baseline

These rules apply before the assistant reads email/calendar content for decisions and before any external action can be proposed:

- Treat all email bodies, calendar descriptions, attachments, Telegram messages, web content, and tool output as untrusted data.
- Do not let untrusted content directly become instructions, tool names, tool arguments, recipients, calendar attendees, labels, or policies.
- Normalize and sanitize HTML before model use: remove scripts, remote tracking elements, hidden text, zero-size text, display-none content, invisible Unicode, confusing directionality markers, and suspicious metadata.
- Run a quarantined extraction step that produces typed fields such as intent, proposed recipients, dates, commitments, risk flags, and source spans.
- Validate extracted fields with deterministic checks before model planning or action proposal creation.
- Never send, forward, invite, export, or disclose data to a recipient introduced only by untrusted content unless the user hard-confirms that recipient.
- Run egress checks for every action that sends content outside the user's connected accounts or introduces a new recipient/attendee.
- Keep tool execution separate from model reasoning. The model may propose structured actions; `ActionPolicyEngine` and `action-executor` decide whether execution is allowed.
- Use append-only, tamper-evident audit events for action proposals, approvals, policy decisions, execution attempts, and derived-from content references.

## Action State Machine

Every external action is represented as a typed OneBrain action record:

1. `proposed`: created from trusted user intent or sanitized/extracted source content.
2. `needs_review`: blocked pending user review, edit, or missing context.
3. `approved`: approved by the user from web, Telegram, or another verified channel when allowed by the action's risk tier.
4. `executing`: claimed by `action-executor` with an idempotency key.
5. `executed`: provider confirms the action completed.
6. `failed`: provider rejected or timed out after retries.
7. `cancelled`: user or policy cancelled the action.

The same action can be shown in web and Telegram, but approval must dedupe to one state transition. Execution must be idempotent so double-clicks, retries, and duplicate webhook callbacks do not double-send or double-create events.

External side effects are emitted through a transactional outbox. The action state transition and outbox intent must commit in the same Postgres transaction. A worker leases the outbox row, rechecks policy, calls the provider with an idempotency key, records the provider response, and marks the outbox row delivered or failed. If the process crashes after approval but before provider execution, the outbox row remains replayable. If the provider call is retried, the idempotency key prevents duplicate sends/events where the provider supports it, and internal dedup prevents the assistant from issuing the same logical action twice.

## Sync Architecture

Use push-first sync for low latency, but assume notifications can expire, fail, or be dropped. Every provider connector needs cursor state, renewal jobs, and reconciliation.

- Gmail: use Gmail `watch` with Pub/Sub where available, store the returned `historyId`, renew watches before expiration, and use `history.list` for incremental reconciliation. If the history ID is invalid or too old, run a full mailbox resync for the relevant scope.
- Google Calendar: use watch channels for changed calendar resources, store `syncToken` per calendar, run incremental sync, and perform full resync when the token expires.
- Microsoft 365 mail/calendar: use Microsoft Graph change notifications for push, renew subscriptions before expiration, store subscription state, and use delta queries/delta links for reconciliation.
- Polling fallback: every connector supports provider-aware polling for test mode, missed notifications, expired subscriptions, webhooks that cannot be exposed, and providers without push support.
- Sync jobs are idempotent and account-scoped. They must handle rate limits, backoff, retries, cursor corruption, duplicate provider events, and partial failure.

## Cost, Quota, And Model Tiering

Email/calendar assistants can become expensive quickly, so cost control is an architecture requirement:

- Do not run full LLM drafting on every email.
- Use cheap classification, rules, embeddings, or small models for first-pass triage.
- Use stronger models only for drafting, complex scheduling, ambiguous intent, sensitive summarization, and user-requested reasoning.
- Cache thread summaries, sender reputation, contact context, recipient tone profiles, and stable calendar insights.
- Batch low-urgency overnight triage on durable jobs.
- Keep LLM calls out of synchronous request paths wherever possible.
- Track per-user and per-provider quota usage for Google, Microsoft Graph, Telegram, Gemini, and other AI providers.
- Degrade gracefully when rate-limited: show partial briefs, stale timestamps, retry status, and provider-specific recovery actions.

## UX Plan

### One-Click UX Rule

The daily product should be centered on `Today`.

Users should not need to navigate through multiple screens to understand what matters now. The default screen should answer:

- What is next?
- What are my top priorities?
- What is risky today?
- What am I waiting on?
- Who is waiting on me?
- What needs approval?
- What should I do now?

Primary daily actions should be one click:

- Approve.
- Dismiss.
- Snooze.
- Edit.
- Ask why.
- Open details.
- Start voice/text assistant.

Two clicks are acceptable for exceptions: editing content, sensitive approvals, settings, OAuth/provider management, privacy choices, and detailed drilldowns.

### Daily Navigation Model

Use a simple daily navigation model:

- `Today`: the home base and default screen.
- `Inbox`: drilldown for triage, drafts, and reply-needed work.
- `Calendar`: drilldown for free slots, conflicts, meeting prep, and focus blocks.
- `Follow-Ups`: drilldown for waiting-on, delegated, overdue, and due-soon work.
- `Settings`: hidden behind profile/settings, not part of the daily working path.

The assistant input should be persistent: a visible text/voice control is always available from the main screens. `Assistant` can exist as a full screen for deeper conversation, but users should not have to visit it to ask a quick question.

### Settings Information Architecture

Settings is powerful, but it must not feel like a technical control panel. Group settings by what the user wants to accomplish, with global search as the first interaction.

Top-level settings groups:

- `Connect my work`: email accounts, calendars, Telegram, voice, providers, OAuth scopes, reconnect, disconnect.
- `Decide what you tell me`: notification categories, Telegram content level, quiet hours, brief timing, meeting prep timing, urgent/VIP rules.
- `Control what you can do`: autonomy levels, approval rules, action risk tiers, draft/send permissions, calendar-write permissions.
- `Protect my data`: privacy, retention, provider/GDPR disclosure, data residency mode, sensitive content rules, export/delete options.
- `Make it feel right`: language, timezone, work hours, voice, tone, assistant style, work/private boundaries.
- `Fix connection problems`: provider health, last sync, expired OAuth grants, webhook status, failed jobs, retry/reconnect actions.

Settings search should understand plain-language queries and synonyms such as "stop Telegram", "change morning brief", "disconnect Gmail", "less notifications", "German", "voice", "private emails", and "can it send emails?"

Settings should use progressive disclosure:

- Show the common decision first in plain language.
- Show technical details only behind "details", "advanced", or provider-specific drilldowns.
- Link directly from cards to the related setting, such as "change this notification rule" or "make this person VIP".
- Keep daily settings changes one click away from the card that triggered them.

The goal is that a business owner can change assistant behavior without knowing whether the setting belongs to Telegram, OAuth, LLM, privacy, scheduler, or OneBrain.

### Personal Assistant Operating Model

The assistant should follow a daily and weekly rhythm:

- Morning: summarize the day, risks, priorities, and recommendations.
- Before meetings: show who is involved, why the meeting matters, prior context, open loops, and likely action items.
- During the day: interrupt only for urgent/VIP messages, schedule conflicts, blocked work, approvals, or user-selected categories.
- After meetings: propose follow-ups, drafts, reminders, and task updates.
- End of day: summarize what moved, what is still waiting, and what should be first tomorrow.
- Weekly: highlight stuck items, overloaded days, neglected clients/projects, recurring distractions, and cleanup recommendations.

The assistant should be calm and opinionated. It should not simply list everything. It should reduce the user's decision load.

### Priority Engine

The assistant should score and explain priority using:

- VIP/client/team relationship.
- Deadline and calendar proximity.
- Sender importance.
- Revenue or business impact where known.
- User-stated goals and current projects.
- Waiting time and overdue status.
- Whether the user is blocking someone else.
- Meeting relevance.
- Confidence level and uncertainty.

Every high-priority recommendation should have a short explanation, such as: "High priority because it is from a VIP client, due today, and blocks tomorrow's meeting."

### Goals, Projects, And Weekly Intentions

The assistant needs lightweight context beyond email and calendar:

- This week's focus.
- Active projects.
- Important clients or relationships.
- Protected focus blocks.
- Delegated responsibilities.
- Topics to ignore or deprioritize.
- Work/private boundaries.

The user should be able to set or correct these quickly from Today, Telegram, or the assistant input.

### Calendar Focus-Time Strategy

Calendar planning should protect the user's work capacity, not only fill empty time.

The assistant should track:

- Protected focus blocks.
- Preferred deep-work hours.
- Meeting density.
- Back-to-back meeting risk.
- Context switching.
- Prep and follow-up time.
- Lunch, travel, and recovery buffers where relevant.
- After-hours pressure.
- Deadline proximity.
- Low-priority movable work.

Calendar recommendations should include:

- "Protect this block" suggestions when the week lacks deep-work time.
- "This day is overloaded" warnings with the reason.
- Suggested movement of low-priority work away from crowded days.
- Prep and follow-up holds around important meetings.
- Batching suggestions for admin work, email review, and small tasks.
- Free-slot suggestions that explain why a slot is good, not just that it is empty.

The assistant must not auto-move external meetings in V1. It can propose moves, draft reschedule messages, create private holds, or suggest blocks, but external calendar writes still follow the approval/action system.

OneBrain should store durable focus preferences and planning feedback, while assistant Postgres stores operational planning jobs, pending calendar proposals, and execution state.

### Relationship Intelligence

A good personal assistant remembers people.

OneBrain should store relationship intelligence for:

- VIPs.
- Clients.
- Internal teammates.
- Private/family contacts.
- Preferred tone.
- Last interaction.
- Open loops.
- Usual meeting style.
- Follow-up expectations.
- Account/source context.

This should drive priority, drafts, meeting prep, briefings, and notification decisions.

### Learning And Feedback Loop

Every important card should teach the assistant with lightweight feedback:

- Good suggestion.
- Not important.
- Never notify me about this.
- This person is VIP.
- Use a shorter tone.
- Remind me later.
- This belongs to private life, not work.
- This project/client matters this week.

Feedback should become durable OneBrain memory where policy allows it. The assistant should improve through these signals before any complex model fine-tuning is considered.

### Today Screen Operating Model

`Today` should be organized for scanning and action:

- Now/next: next meeting or task.
- Top 3 priorities.
- Calendar risks and free-slot opportunities.
- Waiting on me.
- Waiting on others.
- Drafts and approvals.
- One proactive suggestion.
- Provider health only when it affects trust.
- Persistent text/voice assistant control.

Avoid turning Today into a long dashboard. If there are many items, show the few that matter and link to the drilldown.

### Approval Card UX Rules

Every approval card should answer:

- What will happen?
- Which account will act?
- Who receives it or is affected?
- Why did the assistant suggest it?
- What source created it?
- What changed if the user edited it?
- Is it reversible?
- Is it safe for Telegram approval or web/fresh-auth only?

For high-risk actions, the approved content must be snapshotted. If the content, recipients, attendees, sending account, or key fields change after approval, approval resets to `needs_review`.

Approval buttons should match risk:

- Low/medium risk: approve, edit, dismiss, remind later.
- High risk: review/send, edit, dismiss, remind later. Do not make "approve" the dominant default.

- Main surface: `#fff`.
- Secondary surface: `#f1ece2`.
- Ink: `#16191e`.

- Prioritize dense, readable layouts over marketing-style hero sections.

- Typography should use a clean UI face for product work, with any brand/display face used sparingly for onboarding or major section titles only.

### Voice UX Rules

- Start with push-to-talk.
- Keep spoken answers short.
- Always show the artifact behind the spoken answer.
- Let the user interrupt or cancel.
- Never read sensitive content aloud unless explicitly requested.
- Store transcript provenance in OneBrain.
- Support English and German as first-class target languages.
- In MVP, voice can answer questions and create proposals, but cannot directly execute irreversible external actions.

### Telegram UX Rules

- Telegram is a private chat with the assistant, not a public channel.
- Messages are short, useful, and actionable.
- Morning briefs and important updates can arrive in Telegram by default.
- Approval prompts use simple choices: approve, edit on web, dismiss, remind later.
- Sensitive content categories are opt-in per user and per space.
- The user can pause notifications or change quiet hours from Telegram.
- Telegram messages link back to the matching web screen for deeper review.
- Inbound Telegram messages are stored in OneBrain with channel provenance.
- Telegram callbacks must reference existing `ActionStore` records and cannot execute free-form actions from message text.
- Telegram is the first required channel, but implementation must stay behind `NotificationChannel`.

### Web And App-Ready Rules

- Build the first interface as a web app.
- Use mobile-first responsive layouts from the beginning.
- Avoid desktop-only interactions.
- Keep voice, notifications, file/audio capture, and OAuth callbacks behind abstractions that can work in Android/iOS later.
- Prefer PWA-friendly patterns where they do not slow V1.
- Keep a future Capacitor-style native wrapper path open, but do not build native apps in V1.
- Use token-based auth compatible with a native shell, not server-only browser sessions as the only auth model.
- Do not rely on web-only local storage for core credentials, sync state, or durable user data.
- Design notification permissions, deep links, action callbacks, audio capture, and file capture so a native wrapper can adopt the same backend contracts.

## Resolved Product Decisions

### Architecture And Language

Follow OneBrain's current hybrid architecture.

- Backend/API: Python/FastAPI.
- Worker: Python, same backend codebase and domain modules as the API.
- Frontend: Next.js/React/TypeScript.
- Type boundary: FastAPI OpenAPI schema exported into typed TypeScript clients.
- Runtime shape: modular monolith with two backend entrypoints, not microservices.
- Operational state: Postgres.
- Queue/scheduler/locks: Redis.
- Durable business data/memory/audit-of-record: OneBrain.

Do not use TypeScript/NestJS for the assistant backend unless OneBrain itself moves backend authority to TypeScript. The value of matching OneBrain's Python security, policy, data, and retrieval core is higher than a single-language TypeScript stack for this product.

The API should be thin. It handles auth, OAuth, webhooks, settings, approvals, and enqueueing. The worker performs the real assistant work through durable jobs and transactional outbox execution.

### One-Click Personal Assistant UX

The assistant should feel like a calm business personal assistant, not a dashboard full of modules.

Decision:

- `Today` is the default home screen and the center of daily work.
- Main daily tasks should be reachable in one click.
- Two clicks are acceptable only for details, editing, sensitive approvals, provider/account management, privacy settings, or exceptions.
- The assistant input is persistent across main screens.
- `Assistant` can be a full screen for deeper work, but quick questions should not require navigation.
- Daily navigation should be `Today`, `Inbox`, `Calendar`, and `Follow-Ups`, with `Settings` behind the profile/settings area.

### Settings Complexity Decision

Settings should be organized around plain user intent.

Decision:

- Settings is searchable by default.
- Settings groups are `Connect my work`, `Decide what you tell me`, `Control what you can do`, `Protect my data`, `Make it feel right`, and `Fix connection problems`.
- Technical categories such as OAuth, scopes, provider adapters, scheduler, LLM provider, and notification channel should appear only inside relevant details or advanced views.
- Every important card can deep-link to the exact setting that changes similar future behavior.
- Settings copy should use user language: "Telegram updates", "morning brief", "can send emails", "private content", "work hours", "disconnect account", not internal subsystem names.
- Common changes should take one click from the relevant card or two clicks from the Settings home.

This keeps a sophisticated system understandable for business owners who need control but do not want to manage infrastructure.

### Visual Design System Decision

The assistant UI should use the assad-dar.de brand foundation while feeling like a serious work tool.

Decision:

- Main and large surfaces use light backgrounds only.
- Dark colors are used for text, borders, small badges, and subtle emphasis, not for big panels or main screens.
- Use the brand-derived palette from assad-dar.de: `#f7f5f1`, `#fff`, `#f1ece2`, `#16191e`, `#454c57`, `#5f6671`, `#a66e2f`, `#8c5a24`, `#3e5573`, `#1f7a4d`, `#b4453e`, `#ddd6ca`.
- Keep the interface dense but calm: strong hierarchy, compact spacing, hairline separation, little decoration, and excellent scanning.
- Today, approval cards, onboarding, settings, degraded states, and Telegram setup should be designed as first-class surfaces before implementation.
- Status colors must be functional. Critical means risk or failure; success means complete or healthy; copper means brand/primary emphasis; slate means calm information.
- The product should not look like a marketing page, dark AI dashboard, or decorative assistant demo.

### Calendar Focus-Time Decision

Calendar planning should protect attention and energy.

Decision:

- Calendar Plan must show focus protection, overload risk, move candidates, prep/follow-up buffers, and free-slot quality.
- Free slots are ranked by usefulness, not only by duration.
- The assistant should flag overloaded days using explainable signals: too many meetings, back-to-back meetings, no focus time, no recovery/lunch buffer, after-hours pressure, urgent deadlines, or excessive context switching.
- The assistant should propose moving low-priority work away from overloaded days, batching admin work, and preserving deep-work blocks.
- External calendar changes remain proposal-first in V1. The assistant can draft, hold, suggest, or ask for approval; it cannot silently reschedule external meetings.
- User feedback on focus suggestions becomes OneBrain memory: "protect mornings", "Fridays are admin", "never move client calls", "prefer deep work after lunch", and similar signals.

### Personal Assistant Routines

The assistant should run on a predictable work rhythm:

- Morning brief.
- Pre-meeting prep.
- During-day interruption filtering.
- After-meeting follow-up extraction.
- End-of-day recap.
- Weekly review.

The assistant should protect attention. It should interrupt for urgent, VIP, conflict, blocked-work, and approval items, not for every new email.

### Priority, Goals, And Relationship Intelligence

The assistant needs durable context about what matters to the user.

Use OneBrain for:

- Weekly intentions.
- Active projects.
- Important clients.
- Protected focus blocks.
- Delegated responsibilities.
- Work/private boundaries.
- VIPs, clients, internal teammates, private contacts, preferred tone, last interaction, open loops, and follow-up expectations.

Priority should be explainable. The user should see why something is high priority, and should be able to correct it in one click.

### Teaching And Feedback Loop

The assistant should learn from explicit user feedback before any complex model tuning.

Every important card should support lightweight feedback:

- Good suggestion.
- Not important.
- Never notify me about this.
- This person is VIP.
- Use a shorter tone.
- Remind me later.
- This belongs to private life, not work.
- This project/client matters this week.

These signals become OneBrain memories where policy allows it.

### Exact Approved Snapshot Decision

For external email/calendar writes, the assistant must execute the exact approved snapshot.

- Snapshot approved content, recipients/attendees, sending account, calendar account, key fields, and content hash.
- If any important field changes after approval, reset the action to `needs_review`.
- Provider execution uses the approved snapshot only.
- The worker rechecks recipients/attendees and policy immediately before execution.
- A regenerated draft is never silently substituted for an approved draft.

### Phase Ordering Corrections

Telegram and connector capability can be built early, but user-facing risky workflows wait for the approval system.

- Phase 2 Telegram builds binding, delivery, preferences, simple commands, callback plumbing, and safe test approvals.
- Phase 3 email/calendar builds connectors, read-only sync, OAuth, provider health, and draft/event-draft capability behind interfaces.
- User-facing drafts, scheduling proposals, and real external action approvals belong in Phase 6 after approval cards, risk tiers, exact snapshots, and outbox execution are active.

### Data Ownership Matrix

Use a strict data ownership matrix so Postgres does not become a second OneBrain.

| Entity | Assistant Postgres Owns | OneBrain Owns |
|---|---|---|
| User/account/space identity | Local cached IDs only when needed for job execution | Source of truth for user, account, space, permissions, and purpose scope |
| OAuth refresh tokens and Telegram bot tokens | Encrypted values or secret references, rotation state, revocation state | Secret references, granted scopes, ownership, purpose, and audit metadata |
| Provider account metadata | Operational provider account IDs, webhook/subscription state, sync health | Durable connected-account facts, user-visible provenance, scope grants |
| Sync cursors, history IDs, delta tokens | Operational cursor values, lease state, retry/reconciliation state | Cursor references, reconciliation summaries, provider-health facts |
| Email/calendar content | Short-lived processing cache where needed, never a memory source of truth | Durable captured messages/events, summaries, extracted facts, provenance, permissions, retention |
| Telegram inbound/outbound messages | Delivery state, callback state, retries, deduplication | Durable channel provenance, conversation memory, notification facts |
| Action state | State machine, leases, retries, outbox rows, idempotency keys | Action proposal, approval fact, source references, final execution fact, policy decision, audit-of-record |
| Provider health | Operational retry/debug state | User-visible degradation facts and audit-relevant provider events |
| Feedback | Pending processing state | Durable assistant feedback, correction, and learning signal |

Default rule: if it is needed to execute, retry, lock, or recover the product, it lives in assistant Postgres. If it is business memory, user knowledge, permissioned content, provenance, privacy, or audit-of-record, it lives in OneBrain.

### Auth And Session Design

OneBrain should be the identity authority. The assistant should not invent a separate user system.

- Web login resolves a OneBrain user, account, and space.
- The assistant API validates OneBrain user/account/space scope before every read, proposal, approval, and action.
- The assistant stores only the session and operational references it needs to serve the product.
- Telegram binding must be initiated from an authenticated web session.
- Telegram binding maps one private chat to one verified OneBrain user, with explicit account/space selection when needed.
- Telegram approval callbacks must reference existing action IDs and verified binding state; free-form Telegram text cannot approve or execute arbitrary actions.
- High-risk approvals require web approval or fresh-auth confirmation, not Telegram-only approval by default.
- Future mobile auth uses token-based auth compatible with iOS/Android shells, not browser cookies only.
- Session revocation, logout, password change, disabled user, or disconnected provider must invalidate assistant-side sessions and risky pending approvals.
- The web app must use CSRF/origin protections, secure cookies where applicable, CSP, and no durable credential storage in browser local storage.

### Secret Encryption And Rotation Strategy

Use envelope encryption from day one.

- Store encrypted OAuth refresh tokens and Telegram bot tokens in assistant Postgres, or store secret references if a managed secret store is available.
- Store secret references, scope metadata, purpose, ownership, and audit metadata in OneBrain.
- Use a master key from Railway environment variables for MVP.
- Keep the `SecretProvider` interface compatible with later KMS, cloud secret manager, or self-hosted secret manager deployments.
- Prefer per-user or per-account data keys where practical.
- Track secret version, created time, last used time, rotation time, and revoked time.
- Revoke provider tokens on disconnect.
- Rotate Telegram bot tokens by guiding the user through BotFather token regeneration and reconnect.
- Redact tokens, cookies, authorization headers, webhook secrets, email addresses where possible, and sensitive content from logs/traces/errors.
- Backup/restore must include encrypted secrets and the key-management instructions needed to recover them safely.

### Action Risk Tiers And Approval Rules

Use explicit action risk tiers across web, Telegram, voice, and workers.

Low-risk actions:

- Create local reminders.
- Create internal follow-up tasks.
- Add local assistant labels/categories.
- Snooze assistant notifications.
- Save preference changes that do not expose data externally.

Low-risk actions can be approved from Telegram, web, or voice-created proposal cards where the user intent is clear.

Medium-risk actions:

- Create email drafts without sending.
- Create calendar holds or prep blocks visible only to the user.
- Update internal assistant tasks.
- Send non-sensitive Telegram reminders to the bound user.

Medium-risk actions can be approved from Telegram when the approval card clearly shows account, affected object, summary, and changed fields.

High-risk actions:

- Send email.
- Forward email.
- Add BCC/CC recipients.
- Delete email or calendar events.
- Invite external attendees.
- Change or cancel external meetings.
- Export, disclose, or transmit content outside connected user accounts.
- Act on a new recipient, attendee, Reply-To, distribution list, or address introduced only by untrusted content.

High-risk actions require web approval or fresh-auth hard confirmation by default. Telegram can notify the user and deep-link to the web approval screen, but should not be the final approval channel unless the user later enables a narrow, explicit policy.

Approval cards must show:

- Action type and risk tier.
- Sending account or calendar account.
- Recipients/attendees, including new or external identities.
- Source message/event and source excerpt.
- Changed fields.
- Sensitive-content flags.
- Why approval is required.
- Whether the action is reversible.

The UI must never make "approve" the visually dominant default for high-risk actions.

### Queue, Outbox, And Worker Recovery Design

Postgres is the durable source of truth for jobs, outbox rows, actions, and recovery. Redis is for wakeups, fast queueing, distributed locks, delayed jobs, and retries.

Required operational states:

- Job: `queued`, `leased`, `running`, `succeeded`, `retry_wait`, `failed`, `dead_lettered`, `cancelled`.
- Outbox row: `pending`, `leased`, `delivered`, `retry_wait`, `failed`, `dead_lettered`, `cancelled`.
- Action: `proposed`, `needs_review`, `approved`, `executing`, `executed`, `failed`, `cancelled`.

Worker rules:

- Every job and outbox row has an account/user scope, idempotency key, lease owner, lease expiry, retry count, next run time, and correlation ID.
- Workers renew leases while running long jobs.
- Expired leases can be reclaimed.
- Retries use bounded exponential backoff.
- Repeated failures move to dead-letter with a user/operator-visible reason.
- External provider calls use idempotency keys where providers support them and internal deduplication where they do not.
- If Redis restarts, workers can rebuild runnable work from Postgres.
- If a worker crashes after approval but before provider execution, the outbox row remains pending/retryable.
- If OneBrain audit/provenance cannot be written, high-risk action execution is blocked.

### Prompt-Injection Test Matrix

Security tests are Phase 0/Phase 3 requirements, not late hardening.

Test fixtures must include:

- Hidden HTML text.
- Zero-size text.
- White-on-white text.
- CSS `display:none`, `visibility:hidden`, and off-screen positioning.
- Invisible Unicode.
- Unicode directionality markers.
- Quoted/forwarded email chain injection.
- Attachment text injection.
- Calendar invite description/location/attendee injection.
- Fake Reply-To and From display-name spoofing.
- BCC/CC insertion attempts.
- Distribution-list expansion and external-recipient ambiguity.
- "Ignore previous instructions" payloads.
- "Forward this to attacker@example.com" hidden or indirect exfiltration attempts.
- Tool-name or JSON-shaped payloads embedded in email content.

Required pass condition: no email/calendar-derived tool call, draft, recipient, attendee, label, export, or provider write can be created from raw content alone. The content must be sanitized, extracted into typed fields, validated, policy-checked, and presented as an action proposal where required.

Google launch work:

- Decide exact Gmail scopes by feature.
- Decide exact Google Calendar scopes by feature.
- Track restricted/sensitive scope verification requirements.
- Prepare OAuth app branding, privacy policy, support contact, and demo/video materials for review.
- Support test users/internal mode during early Railway testing.

Microsoft launch work:

- Decide exact Microsoft Graph delegated permissions by feature.
- Support business tenants where admin consent may be required.

- Handle revoked tokens, expired grants, reconnect, and disconnect.

Cross-provider requirements:

- Users can start read-only, then upgrade to draft/send/calendar-write scopes later.
- The settings UI must show connected accounts, granted scopes, last sync, provider health, and disconnect/revoke controls.

### MVP Onboarding Flow

The first 10 minutes should be calm, guided, and useful.

1. User signs in and resolves OneBrain account/space.
2. User chooses language, timezone, workday hours, and quiet hours.
3. User connects Telegram through the BotFather tutorial.
4. User verifies the private Telegram chat with a test notification.
5. User connects Google and/or Microsoft read-only email/calendar.
6. User reviews provider/GDPR disclosure and Telegram sensitive-content defaults.
7. Assistant syncs a small safe sample and shows progress.
8. User sees the first Today brief with source/account labels.
9. User sees one follow-up or calendar insight if available.
10. User is asked for optional draft/send/calendar-write scope upgrades only when trying an action that needs them.

The first-session win is not "connect everything." It is: "I understand your day, here are the top things, and I will not act externally without approval."

### Provider And GDPR Disclosure UX

Ship a simple "Data And Providers" settings screen in MVP.

It must show:

- OneBrain's role as durable business data, memory, permissions, retrieval, and audit-of-record.
- Assistant Postgres/Redis role as operational state.
- Railway hosting role for the MVP deployment.
- Google and Microsoft role for email/calendar data.
- Telegram role for private assistant chat and push-style updates.
- Gemini role for initial speed-optimized voice/AI processing.
- Which provider categories may process data outside the EU.
- Which notification categories are enabled.
- Whether Telegram receives summaries only or full sensitive content.
- Whether voice is enabled and what content may be transcribed/spoken.
- Current self-hosted/EU mode status.

Defaults:

- Telegram sends short summaries and action prompts only.
- Full email bodies, private notes, confidential customer content, and sensitive business content require explicit opt-in by category or space.
- Provider health and data residency warnings should be understandable, not legalistic.

### OneBrain Downtime And Audit Availability Policy

If OneBrain is unavailable, the assistant enters degraded mode.

Allowed in degraded mode:

- Show cached read-only UI with stale-data warnings.
- Receive provider webhooks and queue safe reconciliation jobs if policy allows.
- Keep operational retries and provider health checks running.
- Let the user view already-synced operational status.

Blocked in degraded mode:

- External sends.
- Email forwards.
- Deletes.
- External calendar writes.
- Sensitive content exports.
- Autonomy actions.
- Any action requiring new OneBrain audit/provenance writes.

Recovery behavior:

- Revalidate pending high-risk actions after OneBrain returns.
- Write delayed audit/provenance facts where allowed.
- Show the user which briefs/actions were partial or delayed.
- Never silently execute a high-risk action that was approved while audit/provenance was unavailable.

### Email And Calendar Providers

Start with both Google and Microsoft:

- Google Workspace.
- Private Gmail.
- Google Calendar.
- Microsoft 365 Outlook.
- Microsoft 365 Calendar.

The connector model must remain open so common providers can be added later. This includes adapter paths for IMAP/SMTP, CalDAV-compatible calendars, iCloud, Yahoo, and other common email/calendar providers where their APIs and permissions allow it.

### OAuth Scope Strategy

Start with least-privilege delegated permissions and explicit user-facing scope upgrades.

Initial read/sync tier:

- Gmail: `gmail.readonly` where body access is needed; `gmail.metadata` only for metadata-only experiments.
- Google Calendar: calendar read scopes for availability and event sync.
- Microsoft Graph: delegated `Mail.Read` and `Calendars.Read` or narrower supported read scopes where available.

Draft/write tier:

- Gmail: draft/compose-capable scopes only when the user enables draft creation or send-after-approval.
- Google Calendar: calendar write scopes only when the user enables event creation/update after approval.
- Microsoft Graph: `Mail.ReadWrite`, `Mail.Send`, and `Calendars.ReadWrite` only when matching approved capabilities are enabled.

The UI should show which features require additional scopes before requesting them. Admin consent and Google verification requirements must be tracked as launch dependencies.

### Sync Decision

Use push-first sync with reconciliation fallback.

- Gmail uses `watch` plus Pub/Sub where practical, stores `historyId`, renews before expiration, and reconciles with `history.list`.
- Google Calendar uses watch channels where practical, stores `syncToken` per calendar, and full-resyncs on expired tokens.
- Microsoft 365 uses Graph change notifications, subscription renewal, and delta queries/delta links for mail and calendar reconciliation.
- Polling remains available for local testing, missed notifications, providers without push support, expired subscriptions, and operational recovery.

### Prompt-Injection And Egress Decision

Prompt-injection defense is a foundation requirement and a Phase 6 prerequisite for drafts/scheduling.

The assistant must treat external content as data only. It must sanitize/normalize content, extract structured facts in quarantine, validate fields deterministically, and block unsafe egress. Any send, forward, invite, export, or disclosure to a recipient introduced only by external content requires hard user confirmation.

### Action Execution Decision

Every approval-driven operation uses a typed action record with a state machine and idempotency key. Web and Telegram can both display the same action, but Telegram can final-approve only actions eligible under the risk-tier policy. Only one approval transition may reach `executing`. No provider write happens from a raw chat message, email body, or model output alone.

### Cost And Rate-Limit Decision

Use model tiering from the beginning. Cheap classification, rules, embeddings, and cached summaries handle routine triage. Stronger models handle drafts, ambiguous reasoning, and user-requested planning. Queue background jobs for batch work, keep LLM calls out of request paths where practical, and track quota/cost for Google, Microsoft, Telegram, Gemini, and other providers.

### Telegram Bot Setup

The user supplies their own Telegram bot.

The product should include a quick BotFather tutorial:

1. Open Telegram and talk to BotFather.
2. Create a new bot.
3. Copy the bot token.
4. Paste the token into the web app.
5. Send a verification message to the bot.
6. Confirm the private chat binding in the web app.
7. Send a test notification.

The bot identity/name is whatever the user creates in BotFather. The product does not need a shared global bot for MVP.

### Telegram In MVP

Telegram is required in the MVP. It should be part of onboarding, not hidden as an optional advanced integration.

The user should still control what Telegram sends. The MVP should ship with sensible defaults and a simple preferences screen.

Default Telegram update categories:

- Morning brief.
- Calendar conflicts and important schedule changes.
- Upcoming meeting prep reminders.
- Follow-ups due or overdue.
- Action approvals waiting.
- Urgent/VIP email alerts.
- System, security, and connection issues.

Sensitive message content should be configurable. By default, Telegram can show short summaries and action prompts, while full email bodies, private notes, and sensitive business content require explicit opt-in by category or space.

### Voice Provider

Optimize first for speed. Use Gemini for the first voice implementation, behind the `SpeechToTextProvider` and `TextToSpeechProvider` interfaces so the provider can change later for cost, EU processing, quality, or customer-specific compliance.

MVP voice is intentionally constrained: it can answer, summarize, and create proposal cards, but irreversible external actions still require approval through the action system.

### Languages

German and English are required from MVP. Users can switch language in settings, and the assistant should remember language preference per user. The UI and assistant responses should support both languages directly.

### First User Surface

The first user surface is both mobile web and desktop web. Mobile web cannot be a reduced version: briefings, Telegram setup, voice, approvals, drafts, follow-ups, and calendar review must all work on mobile-sized screens.

### Future Android And iOS Path

Do not choose the native wrapper yet. Keep the architecture app-ready and decide later based on feature needs.

Current bias:

- Use Capacitor if the mobile web/PWA experience covers most needs and we mainly need native packaging, push, share targets, and device APIs.
- Use React Native or another native path if voice, background behavior, notifications, offline storage, or platform-specific integrations require deeper native control.

The V1 rule is to avoid web decisions that block either path.

Concrete app-ready constraints for V1:

- Token-based auth that a native shell can use.
- Backend-owned durable state, not browser-only local storage.
- Notification model based on channel adapters and deep-linkable action IDs.
- Audio capture and upload through backend contracts that can be reused by native apps.
- OAuth callback flow that can support browser, PWA, and native redirect patterns.
- No desktop-only core workflows.

### OneBrain Contract Location

Best recommendation: implement OneBrain contracts in the OneBrain repo first, because OneBrain owns durable business data, permissions, retention, audit-of-record, and retrieval. This repo should keep the product-facing requirements and client expectations, but the canonical schema/API contracts belong in OneBrain.

Recommended workflow:

1. Keep this implementation plan as the assistant product spec.
2. Add or update OneBrain contracts in the OneBrain repo.
3. Extend the OneBrain SDK/client.
4. Build the assistant module against those stable OneBrain contracts.
5. Only use temporary mock clients in this repo for UI prototyping, and remove them before real data flows.
